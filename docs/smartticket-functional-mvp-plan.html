Z-API vs Twilio — por que Z-API agora

Z-API — MVP Funcional
  • Sem processo de aprovação — número ativo imediatamente
  • Custo fixo ~R$150–250/mês, sem taxa por mensagem
  • Foco PT-BR, suporte nacional
  • Ideal para validar o produto com cliente piloto rapidamente
  • Interface abstrata no código garante troca sem reescrita
Twilio / Meta Cloud API — Próximo milestone
  • BSP oficial certificado pela Meta — menor risco de bloqueio
  • Documentação excelente, mais estável em produção
  • Suporte a HSM templates para mensagens ativas
  • Meta Business Verification: 3–10 dias úteis
  • Custo por conversa (~US$0,005–0,08) — mais caro em volume

  • Alvo: 400+ exemplos totais, 80+ por categoria (technical_issue, billing_inquiry, refund_request, cancellation_request, product_inquiry)
  • Verificar balanceamento de classes e remover duplicatas

Pipeline PT-BR

  • Estender normalize_text para aceitar language='pt' e carregar stopwords('portuguese')
  • Atualizar run_pipeline para passar language adiante
  • Adicionar env var SMARTTICKET_PIPELINE_LANGUAGE (default 'en')
  • Garantir que nltk_bootstrap.py baixa stopwords de portuguese além de english

Modelo + Testes

  • Retreinar com python -m app.ml.train e validar classification_report
  • Atualizar tests/test_normalizer.py e tests/test_pipeline.py para cobertura PT-BR
  • Atualizar .env.example com SMARTTICKET_PIPELINE_LANGUAGE
Critério de saída: modelo treinado em PT-BR com accuracy aceitável nas 5 categorias, testes passando.
F2 Schema banco + novas rotas da API 3–5 dias · Semana 1–2
Por que antes do WhatsApp: o webhook vai precisar salvar contato e mensagem. Melhor ter o schema pronto e testado antes de integrar o canal.

Banco de dados — tabelas novas

  • Tabela contacts: id (UUID), phone, name, channel, created_at
  • Tabela messages: id, contact_id, ticket_id, direction (inbound/outbound), body, external_id (id do BSP), created_at
  • Extend tickets: adicionar contact_id (FK), channel (whatsapp/api), transições de status (open → in_progress → resolved/escalated)
  • Migration SQL: db/migrations/002_functional_mvp_schema.sql

SQLAlchemy + Repositórios

  • Models: Contact e Message em app/db/models.py
  • contact_repository.py: upsert por telefone
  • message_repository.py: insert, list por ticket

Novas rotas da API

  • GET /tickets/{id} — detalhe do ticket com mensagens
  • PATCH /tickets/{id} — atualizar status e/ou urgency override manual
  • POST /tickets/{id}/reply — atendente envia resposta (persiste + envia via canal)
  • GET /contacts/{id}/tickets — histórico de tickets do contato
  • Atualizar docs/architecture.md e docs/api-contracts.md
Critério de saída: migration aplicada, models criados, rotas funcionando com testes.
F3 Canal WhatsApp (Z-API) 5–7 dias · Semana 2–3
Entrega desta sprint: mensagem enviada pelo cliente no WhatsApp aparece no banco como ticket classificado e priorizado.

Inbound (receber mensagens)

  • Endpoint POST /webhooks/whatsapp com parsing do payload da Z-API
  • Idempotência: verificar se external_id já existe em messages antes de processar (evita duplicatas se Z-API reenviar)
  • Extrair contato (telefone, nome) → upsert em contacts
  • Extrair texto da mensagem → rodar classify_ticket → persistir ticket + message (inbound)

Outbound (enviar mensagens)

  • app/services/channel/zapi_client.py: encapsula chamada HTTP de envio de mensagem da Z-API
  • Interface abstrata app/services/channel/base.py com método send_message(phone, text)
  • Troca de provider via SMARTTICKET_CHANNEL_PROVIDER (zapi/twilio) — Twilio = novo arquivo, zero reescrita

Config + Testes

  • Env vars: ZAPI_INSTANCE_ID, ZAPI_TOKEN, ZAPI_BASE_URL, SMARTTICKET_CHANNEL_PROVIDER em .env.example
  • Testes: parsing do webhook, lógica de idempotência, mock do HTTP de saída
Migração para Twilio: implementar app/services/channel/twilio_client.py contra a mesma interface. Mudar só SMARTTICKET_CHANNEL_PROVIDER=twilio. O resto do código não muda.
Critério de saída: mensagem no WhatsApp → ticket classificado no banco de dados.
F4 LLM + APScheduler worker + priority aging 5–7 dias · Semana 3–4
Entrega desta sprint: ticket entra via WhatsApp, worker processa, LLM responde automaticamente, resposta chega no WhatsApp do cliente.

LLM Integration (OpenAI GPT-4o mini)

  • Completar app/services/llm_service.py: cliente OpenAI com context injection (raw_text + category + score + urgency)
  • Templates de prompt por categoria (PT-BR) em app/services/llm_prompts.py
  • Feature toggle: SMARTTICKET_LLM_ENABLED (default false) — se desligado ou se LLM falhar, ticket vai para fila humana
  • Resposta do LLM salva em messages (direction outbound, source llm)
  • Envio da resposta via Z-API outbound

APScheduler Worker (dentro do FastAPI)

  • Worker configurado no lifespan do FastAPI com intervalo SMARTTICKET_WORKER_INTERVAL_SECONDS (default 30s)
  • Lógica: pega próximo ticket com queue_target: llm e status: open na ordem HIGH → MEDIUM → LOW + FIFO
  • Fluxo: chama LLM → envia resposta → marca ticket resolved
  • Fallback: se LLM falhar → escala para fila humana (queue_target: human, status: open)

Priority Aging

  • Job separado no APScheduler rodando a cada 1 hora
  • Promove tickets MEDIUM + status: open mais antigos que SMARTTICKET_AGING_HOURS_MEDIUM horas (default 4h) para HIGH
  • Registra cada promoção no log

Env vars + Testes

  • Env vars: OPENAI_API_KEY, SMARTTICKET_LLM_ENABLED, SMARTTICKET_WORKER_INTERVAL_SECONDS, SMARTTICKET_AGING_HOURS_MEDIUM
  • Testes: LLM service (OpenAI mockado), rendering de prompt, lógica de aging
Migração futura: APScheduler → RabbitMQ quando dividir o monólito em Ingestion API + Query API. APScheduler não coordena trabalho entre múltiplos processos; RabbitMQ se torna necessário nesse ponto.
Critério de saída: fluxo LLM ponta a ponta funcionando com feature toggle.

Pós-MVP Funcional — Próximo milestone

O que está fora do escopo do MVP Funcional mas já está no roadmap e planejado:

ItemTrigger para começarImpacto
Z-API → Twilio / Meta Cloud API Antes de escalar para mais de 1–2 clientes pagantes Estabilidade de canal, menor risco de bloqueio, templates HSM
APScheduler → RabbitMQ Quando dividir o monólito em Ingestion API + Query API Garantia de entrega, múltiplos workers, dead-letter queue
Next.js UI — iteração Após feedback do piloto WebSocket real-time, dashboard de analytics, melhor UX
Feedback loop Quando houver 200+ tickets reais corrigidos Correções do atendente alimentam retreinamento do modelo
Multi-cliente Quando houver 2+ clientes pagantes Config de categoria e artefatos de modelo por cliente
Split do monólito (Ingestion + Query API) Quando o volume justificar escala independente Cada API escala por demanda, arquitetura final do produto
